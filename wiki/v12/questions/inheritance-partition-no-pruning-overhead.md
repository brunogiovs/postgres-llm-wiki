# Inheritance partition query overhead when no pruning is possible

## Question

In PostgreSQL 12, for a query against a table partitioned with traditional table inheritance, the plan cannot prune any of roughly 300 child tables. The executor visits all 300 children, most return no rows, and the query needs to run under 30 ms. What tunables, settings, or configuration choices can reduce the overhead of that plan, and which ones add overhead?

## Answer

Assumption: "PostgreSQL 12" means the local source checkout pinned to `45b88269a353ad93744772791feb6d01bc7e1e42`, and "inheritance partitioning" means traditional inheritance children, not a declaratively partitioned table.

The hard limit is that no GUC can make a no-pruning inheritance plan stop being a 300-child `Append`. PG 12 expands every traditional-inheritance child into range-table and planner structures, obtains child locks, then runs constraint exclusion for each child; if constraint exclusion cannot prove a child impossible, the child stays live and gets planned [[raw/postgres-12/src/backend/optimizer/util/inherit.c#expand_inherited_rtentry|inherit.c#expand_inherited_rtentry]], [[raw/postgres-12/src/backend/optimizer/path/allpaths.c#set_append_rel_size|allpaths.c#set_append_rel_size]]. At execution, an `Append` without runtime partition-prune state marks every subplan valid, initializes every valid subplan, and iterates subplans until each returns a tuple or EOF [[raw/postgres-12/src/backend/executor/nodeAppend.c#ExecInitAppend|nodeAppend.c#ExecInitAppend]], [[raw/postgres-12/src/backend/executor/nodeAppend.c#ExecAppend|nodeAppend.c#ExecAppend]].

So the practical strategy is:

1. Make pruning/exclusion possible if the query is value-selective.
2. If pruning truly cannot happen, make each child probe cheap and avoid extra planner/executor instrumentation.
3. Treat parallelism and custom planning as measured per-query choices, because they can help one shape and hurt another.

## Reduce Overhead

|---|---|---|---|
| `constraint_exclusion` | Keep `partition` unless testing proves this specific query can never exclude any child; try `on` only when directly named partitions or non-inheritance tables need exclusion. | For inheritance partitioning, `constraint_exclusion=partition` is the default and checks inheritance children / `UNION ALL` subqueries. It omits a child only when query restrictions contradict safe `CHECK` constraints. If there are no base restrictions, or no immutable constraints that can be refuted, it returns "not excluded" and all children remain live [[raw/postgres-12/doc/src/sgml/config.sgml#guc-constraint-exclusion|config.sgml#guc-constraint-exclusion]], [[raw/postgres-12/src/backend/optimizer/util/plancat.c#relation_excluded_by_constraints|plancat.c#relation_excluded_by_constraints]]. | `PGC_USERSET`; session/transaction `SET` needs no restart or reload. Changing the default in config/`ALTER SYSTEM` needs the usual reload for defaults to propagate [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesEnum|guc.c#constraint_exclusion]]. |
| Child `CHECK` constraints and query shape | Best possible fix: make the predicate contradict 299 child constraints. | Traditional inheritance partitioning depends on non-overlapping child `CHECK` constraints. PG 12 docs warn that inheritance partitioning performs poorly when constraint exclusion cannot prune children, and the implementation marks a child dummy only after `relation_excluded_by_constraints()` succeeds [[raw/postgres-12/doc/src/sgml/ddl.sgml#ddl-partitioning-implementation-inheritance|ddl.sgml#inheritance-partitioning]], [[raw/postgres-12/src/backend/optimizer/path/allpaths.c#set_append_rel_size|allpaths.c#set_append_rel_size]]. | Schema/query change, not a GUC. |
| `plan_cache_mode=force_custom_plan` | Test if the query is prepared/parameterized and a generic plan hides the partition key value. | A custom plan is built for a specific execution using the supplied parameter values, while a generic plan is reused. `plan_cache_mode` can force either choice; source policy also chooses custom plans for the first five parameterized executions in `auto` mode [[raw/postgres-12/doc/src/sgml/ref/prepare.sgml#PREPARE|prepare.sgml#custom-and-generic-plans]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]]. | Per-child indexes on the filtering columns | Make surviving child scans cheap. | With inheritance partitioning, PG 12 docs say there is no point defining indexes on the master table and instruct creating indexes on each child. If all 300 children must be visited, indexes can turn "read/filter many rows" into "quickly prove no matching row" for each child, when selectivity supports it [[raw/postgres-12/doc/src/sgml/ddl.sgml#ddl-partitioning-inheritance-example|ddl.sgml#inheritance-example]]. | Schema/index change. Indexes add write/maintenance cost and more paths for the planner to consider. |
| Per-child `ANALYZE` and stats | Keep estimates usable for every child. | PG 12 docs explicitly warn that manual `ANALYZE measurement` processes only the master table for inheritance partitioning, so child tables need their own `ANALYZE` when not relying on autovacuum [[raw/postgres-12/doc/src/sgml/ddl.sgml#ddl-partitioning-inheritance-caveats|ddl.sgml#inheritance-caveats]]. Better child stats help the planner choose index probes where they are actually cheaper. | Maintenance operation, not a restart/reload setting. |
| Planner cost knobs: `random_page_cost`, `seq_page_cost`, `effective_cache_size` | Calibrate only if the planner is choosing seq scans where cached/SSD index probes are actually faster. | Lower `random_page_cost` relative to `seq_page_cost` makes index scans look cheaper; higher `effective_cache_size` also makes index scans more likely. PG 12 docs stress that cost constants are averages for the workload and changing them from a few experiments is risky [[raw/postgres-12/doc/src/sgml/config.sgml#guc-random-page-cost|config.sgml#guc-random-page-cost]], [[raw/postgres-12/doc/src/sgml/config.sgml#guc-effective-cache-size|config.sgml#guc-effective-cache-size]]. | `PGC_USERSET`; session/transaction tests need no restart or reload; config/default changes need reload [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesReal|guc.c#planner-costs]]. |
| `enable_indexscan`, `enable_indexonlyscan`, `enable_bitmapscan` | Leave on. | These booleans allow the planner to choose index and bitmap paths. Disabling them can force 300 child seq scans when indexes would have been the cheap no-row proof [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesBool|guc.c#index-scan-gucs]]. | `PGC_USERSET`; session/transaction `SET` needs no restart or reload. |
| `enable_seqscan=off` | Use only as a diagnostic experiment, not a permanent fix. | PG 12 docs say seq scans cannot be suppressed entirely; turning this off only discourages them if alternatives exist [[raw/postgres-12/doc/src/sgml/config.sgml#guc-enable-seqscan|config.sgml#guc-enable-seqscan]]. | `max_parallel_workers_per_gather=0` and/or `enable_parallel_append=off` | Test disabling for the sub-30 ms hot path. | Parallel query is considered only when `max_parallel_workers_per_gather > 0`; `enable_parallel_append` lets the planner build parallel-aware append paths and can request workers scaled by the number of live children, bounded by `max_parallel_workers_per_gather` [[raw/postgres-12/src/backend/optimizer/plan/planner.c#standard_planner|planner.c#standard_planner]], [[raw/postgres-12/src/backend/optimizer/path/allpaths.c#add_paths_to_append_rel|allpaths.c#add_paths_to_append_rel]]. ## Settings That Add Overhead

| Setting/configuration | Overhead risk | Source-grounded reason |
|---|---|---|
| `constraint_exclusion=on` | Extra planning CPU beyond `partition`. | `on` examines constraints for all tables; docs say this extra planning overhead is noticeable on simple queries and often gives no benefit [[raw/postgres-12/doc/src/sgml/config.sgml#guc-constraint-exclusion|config.sgml#guc-constraint-exclusion]]. |
| `constraint_exclusion=partition` when no child can ever be excluded | Planning-time constraint checks for every inheritance child. | The planner calls `relation_excluded_by_constraints()` for each append child before deciding it remains live [[raw/postgres-12/src/backend/optimizer/path/allpaths.c#set_append_rel_size|allpaths.c#set_append_rel_size]]. Turning it `off` can reduce planning work only for queries where exclusion is known impossible, but it will also prevent useful inheritance exclusion for other queries. |
| `plan_cache_mode=force_custom_plan` | Replanning every execution. | Custom plans use execution parameter values, but the plan-cache policy includes planning cost in custom-plan cost and uses generic plans when they are cheaper in `auto` mode [[raw/postgres-12/src/backend/utils/cache/plancache.c#cached_plan_cost|plancache.c#cached_plan_cost]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]]. |
| `force_generic_plan` for parameterized partition-key predicates | Can lock in all-child plans. | Generic plans are the same across executions, while custom plans are generated with supplied parameter values; `EXPLAIN EXECUTE` shows parameter symbols for generic plans and substituted values for custom plans [[raw/postgres-12/doc/src/sgml/ref/prepare.sgml#PREPARE|prepare.sgml#custom-and-generic-plans]]. Traditional inheritance has no executor-time partition pruning equivalent to declarative partition pruning, so plan-time exclusion matters [[raw/postgres-12/src/backend/partitioning/partprune.c#prune_append_rel_partitions|partprune.c#prune_append_rel_partitions]], [[raw/postgres-12/src/backend/optimizer/plan/createplan.c#create_append_plan|createplan.c#create_append_plan]]. |
| `enable_partition_pruning` | Usually no help for traditional inheritance; do not count it as the fix. | PG 12 docs define this for declaratively partitioned tables. The pruning code asserts a partition scheme and returns all partitions when pruning is disabled or unusable; `create_append_plan()` adds executor prune info only for append paths that record partitioned rels [[raw/postgres-12/doc/src/sgml/config.sgml#guc-enable-partition-pruning|config.sgml#enable_partition_pruning]], [[raw/postgres-12/src/backend/partitioning/partprune.c#prune_append_rel_partitions|partprune.c#prune_append_rel_partitions]], [[raw/postgres-12/src/backend/optimizer/plan/createplan.c#create_append_plan|createplan.c#create_append_plan]]. |
| `enable_partitionwise_join=on`, `enable_partitionwise_aggregate=on` | More planner CPU and memory; no direct benefit for a simple inheritance-parent scan. | PG 12 docs say both are off by default because they can use significantly more CPU and memory during planning, and the planner gate for partitionwise join checks `RELKIND_PARTITIONED_TABLE` [[raw/postgres-12/doc/src/sgml/config.sgml#guc-enable-partitionwise-join|config.sgml#partitionwise-join]], [[raw/postgres-12/src/backend/optimizer/path/allpaths.c#set_append_rel_size|allpaths.c#partitionwise-join-gate]]. |
| Low `jit_above_cost` or `jit=on` for short plans that cross the threshold | Compilation/setup overhead. | The planner sets JIT flags when plan cost exceeds the configured threshold; `jit_above_cost=-1` disables that path [[raw/postgres-12/src/backend/optimizer/plan/planner.c#standard_planner|planner.c#jit-flags]], [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesReal|guc.c#jit_above_cost]]. |
| `track_io_timing=on` | Repeated operating-system time calls around I/O. | PG 12 docs say it is off by default because it repeatedly queries the OS clock and may cause significant overhead; buffer read/write paths conditionally take timestamps when it is on [[raw/postgres-12/doc/src/sgml/config.sgml#guc-track-io-timing|config.sgml#track_io_timing]], [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#ReadBuffer_common|bufmgr.c#read-timing]], [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#FlushBuffer|bufmgr.c#write-timing]]. `PGC_SUSET`: superuser session setting; no restart/reload for session use, reload only to change defaults [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesBool|guc.c#track_io_timing]]. |
| `auto_explain.log_analyze=on`, especially with `auto_explain.log_timing=on` | Per-plan-node instrumentation and repeated clock reads for every sampled statement. | The `auto_explain` docs warn this can have an extremely negative performance impact, and the module installs executor hooks that add timer/row/buffer instrumentation when enabled [[raw/postgres-12/doc/src/sgml/auto-explain.sgml#auto-explain|auto-explain.sgml#log_analyze]], [[raw/postgres-12/contrib/auto_explain/auto_explain.c#explain_ExecutorStart|auto_explain.c#explain_ExecutorStart]]. The custom GUCs are `PGC_SUSET` after the module is loaded [[raw/postgres-12/contrib/auto_explain/auto_explain.c#_PG_init|auto_explain.c#_PG_init]]. |
| `log_planner_stats`, `log_executor_stats`, `log_statement_stats` | Stats collection and logging work on the hot path. | These are off by default `PGC_SUSET` booleans, and `PortalRun()` resets and shows usage when executor stats are enabled [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesBool|guc.c#log-stats-gucs]], [[raw/postgres-12/src/backend/tcop/pquery.c#PortalRun|pquery.c#PortalRun]]. |

## 30 ms Triage Order

For this exact symptom, measure planning time and execution time separately. If planning dominates, test `constraint_exclusion=off` only for this query shape, test `force_generic_plan` only when parameters are irrelevant, and keep partitionwise options off. If execution dominates, check whether every child has a suitable local index and current stats, then test parallel append both ways. If the plan reports JIT, disable it for the session before chasing smaller knobs.

The most likely winning change is still not a cost constant: rewrite the query or partition constraints so the planner can refute most children, or route the query directly to the known child table when the application already knows the partition key. PG 12's own inheritance-partitioning documentation says poor performance is expected when constraint exclusion cannot prune child tables effectively [[raw/postgres-12/doc/src/sgml/ddl.sgml#ddl-partitioning-implementation-inheritance|ddl.sgml#inheritance-partitioning]].

## Open Questions

- The pinned PG 12 source does not determine whether this specific workload can meet 30 ms; that depends on the exact query text, child table sizes, indexes, storage/cache state, prepared-statement behavior, and measured planning/execution split.
- This review did not run a local benchmark with 300 inherited child tables. The recommendations are source-grounded direction-of-effect guidance, not measured numeric settings.
- If the query predicates do not contain any value that can refute child `CHECK` constraints, the source-visible options reduce overhead only at the margins; the structural fixes are fewer children, direct child routing, declarative partitioning where runtime pruning applies, or a query/schema change that makes exclusion possible.
