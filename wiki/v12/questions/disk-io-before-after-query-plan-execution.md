# Disk I/O before/after query planning and execution

## Question

For PostgreSQL 12, what disk I/O can happen before planning, during planning, during execution, and after execution? If the database is running on slow or contended Azure storage and most table pages are already in `shared_buffers`, can query latency still be affected by disk I/O outside the main heap/index reads? How should this differ across common statement types? Also, give a precise analysis of this measurement claim: `track_io_timing` times backend buffer reads and writes around `smgrread` and `smgrwrite`; it does not time shared-buffer hits, metadata opens/lseeks, WAL flushes, or background-process I/O for the foreground query.

## Answer

Assumption: this review is for PostgreSQL 12 at pinned commit `45b88269a353ad93744772791feb6d01bc7e1e42`.

The short answer is: a `shared_buffers` hit avoids PostgreSQL's data-file read syscall for that block, but it does not make the whole statement independent of storage. PostgreSQL can still touch storage through catalog and relcache misses, relation-size probes, temp-file spills, WAL writes and flushes, dirty-buffer eviction, checkpointer/bgwriter writes, DDL file operations, and utility-command work. The simple-query path explicitly separates raw parse, parse analysis/rewrite, planning, portal start/run, and transaction finish; PostgreSQL's own comment notes that parse analysis, rewrite, and planning can access the database, so those phases are not automatically "CPU only" [[raw/postgres-12/src/backend/tcop/postgres.c#exec_simple_query|postgres.c#exec_simple_query]].

### Query Lifecycle I/O

**Raw parse.** In the simple-query path, `pg_parse_query` calls the raw parser to produce raw parse trees [[raw/postgres-12/src/backend/tcop/postgres.c#pg_parse_query|postgres.c#pg_parse_query]]. The database-access risk starts after raw parse: `exec_simple_query` checks for aborted transactions before parse analysis because parse analysis, rewrite, and planning "try to do database accesses" [[raw/postgres-12/src/backend/tcop/postgres.c#exec_simple_query|postgres.c#exec_simple_query]].

**Parse analysis and rewrite.** Name resolution opens referenced relations. `parserOpenTable` calls the table open path, `table_openrv_extended` delegates to relation open, and `relation_openrv_extended` accepts invalidation messages, resolves the relation OID, and opens the relation with the requested lock [[raw/postgres-12/src/backend/parser/parse_relation.c#parserOpenTable|parse_relation.c#parserOpenTable]] [[raw/postgres-12/src/backend/access/table/table.c#table_openrv_extended|table.c#table_openrv_extended]] [[raw/postgres-12/src/backend/access/common/relation.c#relation_openrv_extended|relation.c#relation_openrv_extended]]. A relcache hit can satisfy this from memory, but a relcache miss builds a descriptor, and catalog cache misses open and scan system catalogs through `systable_beginscan` / `systable_getnext` [[raw/postgres-12/src/backend/utils/cache/relcache.c#RelationIdGetRelation|relcache.c#RelationIdGetRelation]] [[raw/postgres-12/src/backend/utils/cache/catcache.c#SearchCatCacheMiss|catcache.c#SearchCatCacheMiss]] [[raw/postgres-12/src/backend/access/index/genam.c#systable_beginscan|genam.c#systable_beginscan]].

**Planning.** Utility commands do not get normal plans; non-utility queries call `planner` through `pg_plan_query` / `pg_plan_queries` [[raw/postgres-12/src/backend/tcop/postgres.c#pg_plan_query|postgres.c#pg_plan_query]] [[raw/postgres-12/src/backend/tcop/postgres.c#pg_plan_queries|postgres.c#pg_plan_queries]]. The planner can probe relation sizes: `estimate_rel_size` calls the table AM size estimator for tables and `RelationGetNumberOfBlocks` for indexes, heap table size estimation calls `RelationGetNumberOfBlocks`, and that eventually reaches storage-manager block counts [[raw/postgres-12/src/backend/optimizer/util/plancat.c#estimate_rel_size|plancat.c#estimate_rel_size]] [[raw/postgres-12/src/backend/access/heap/heapam_handler.c#heapam_estimate_rel_size|heapam_handler.c#heapam_estimate_rel_size]] [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#RelationGetNumberOfBlocksInFork|bufmgr.c#RelationGetNumberOfBlocksInFork]]. In `md.c`, block counts can open relation segments and use file size operations, which means planning may pay metadata/file-operation latency even when user data pages are already cached in shared buffers [[raw/postgres-12/src/backend/storage/smgr/md.c#mdnblocks|md.c#mdnblocks]] [[raw/postgres-12/src/backend/storage/file/fd.c#FileSize|fd.c#FileSize]].

**Executor startup and normal reads.** `PortalRun` reaches `ExecutorRun`, which calls the standard executor and then `ExecutePlan` / `ExecProcNode` through the plan tree [[raw/postgres-12/src/backend/executor/execMain.c#ExecutorRun|execMain.c#ExecutorRun]] [[raw/postgres-12/src/backend/executor/execMain.c#standard_ExecutorRun|execMain.c#standard_ExecutorRun]]. A sequential scan starts a table scan and fetches tuples from the table AM; heap scanning reads heap pages with `ReadBufferExtended` [[raw/postgres-12/src/backend/executor/nodeSeqscan.c#SeqNext|nodeSeqscan.c#SeqNext]] [[raw/postgres-12/src/backend/access/heap/heapam.c#heapgetpage|heapam.c#heapgetpage]]. An index scan fetches index TIDs, then fetches heap tuples when needed [[raw/postgres-12/src/backend/executor/nodeIndexscan.c#IndexNext|nodeIndexscan.c#IndexNext]] [[raw/postgres-12/src/backend/access/index/indexam.c#index_getnext_slot|indexam.c#index_getnext_slot]]. B-tree index pages are also read through the buffer manager [[raw/postgres-12/src/backend/access/nbtree/nbtpage.c#_bt_getbuf|nbtpage.c#_bt_getbuf]].

**Shared-buffer hit versus storage read.** `ReadBuffer_common` first looks up the block in shared buffers. If the block is already present, it returns after recording a buffer hit and does not call the storage manager read path [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#ReadBuffer_common|bufmgr.c#ReadBuffer_common]]. If it misses, it calls `smgrread`, which dispatches to `mdread`, which reads with `FileRead` / `pg_pread` [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrread|smgr.c#smgrread]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdread|md.c#mdread]] [[raw/postgres-12/src/backend/storage/file/fd.c#FileRead|fd.c#FileRead]]. So a warm shared buffer avoids the backend data-file read syscall for that page; an operating-system cache hit only helps after PostgreSQL has missed shared buffers and entered the file-read path.

**Execution writes.** `INSERT` writes a heap tuple, marks the heap buffer dirty, emits heap WAL for logged relations, and then inserts index tuples through `ExecInsertIndexTuples` [[raw/postgres-12/src/backend/executor/nodeModifyTable.c#ExecInsert|nodeModifyTable.c#ExecInsert]] [[raw/postgres-12/src/backend/access/heap/heapam.c#heap_insert|heapam.c#heap_insert]] [[raw/postgres-12/src/backend/executor/execIndexing.c#ExecInsertIndexTuples|execIndexing.c#ExecInsertIndexTuples]]. `UPDATE` reads the old tuple page, may write a new tuple page and TOAST data, can use HOT only when indexed columns do not require index changes, and logs heap update records [[raw/postgres-12/src/backend/access/heap/heapam.c#heap_update|heapam.c#heap_update]] [[raw/postgres-12/src/backend/access/heap/heapam.c#log_heap_update|heapam.c#log_heap_update]]. `DELETE` reads and modifies the heap page, emits heap-delete WAL, may delete TOAST data, and does not remove index entries immediately; VACUUM handles dead index entries later [[raw/postgres-12/src/backend/executor/nodeModifyTable.c#ExecDelete|nodeModifyTable.c#ExecDelete]] [[raw/postgres-12/src/backend/access/heap/heapam.c#heap_delete|heapam.c#heap_delete]].

**SELECT can still dirty pages.** Visibility checks can set commit/abort hint bits through `SetHintBits`. `MarkBufferDirtyHint` marks the page dirty, and it logs a full-page image for the hint-bit update when checksums or `wal_log_hints` require it [[raw/postgres-12/src/backend/access/heap/heapam_visibility.c#SetHintBits|heapam_visibility.c#SetHintBits]] [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#MarkBufferDirtyHint|bufmgr.c#MarkBufferDirtyHint]] [[raw/postgres-12/src/include/access/xlog.h#XLogHintBitIsNeeded|xlog.h#XLogHintBitIsNeeded]]. This is a write side effect of reading, not a guaranteed write for every `SELECT`.

**Temp-file spills.** Sorts stay in memory when small, but large sorts write temporary files; PostgreSQL's tuple sort code uses external sort tapes for that case [[raw/postgres-12/src/backend/utils/sort/tuplesort.c#tuplesort_begin_common|tuplesort.c#tuplesort_begin_common]]. Hash join batching saves tuples to temp `BufFile`s; `BufFile` creates temporary files and reads/writes them with `FileRead` / `FileWrite` [[raw/postgres-12/src/backend/executor/nodeHashjoin.c#ExecHashJoinSaveTuple|nodeHashjoin.c#ExecHashJoinSaveTuple]] [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileCreateTemp|buffile.c#BufFileCreateTemp]] [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileDumpBuffer|buffile.c#BufFileDumpBuffer]] [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileLoadBuffer|buffile.c#BufFileLoadBuffer]]. These temp-file I/Os are outside the main relation-buffer read path, but they still wait on the filesystem and storage device.

**Dirty-buffer writes and fsync.** A backend may have to write a dirty victim buffer before it can reuse that buffer [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#BufferAlloc|bufmgr.c#BufferAlloc]]. `FlushBuffer` writes the page through `smgrwrite`, first flushing WAL up to the page LSN for permanent relations [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#FlushBuffer|bufmgr.c#FlushBuffer]] [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrwrite|smgr.c#smgrwrite]]. `mdwrite` calls `FileWrite` / `pg_pwrite` and registers dirty segments for later fsync unless fsync is skipped or the relation is temporary [[raw/postgres-12/src/backend/storage/smgr/md.c#mdwrite|md.c#mdwrite]] [[raw/postgres-12/src/backend/storage/file/fd.c#FileWrite|fd.c#FileWrite]] [[raw/postgres-12/src/backend/storage/smgr/md.c#register_dirty_segment|md.c#register_dirty_segment]]. The bgwriter and checkpointer also write dirty buffers and absorb fsync requests [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#BgBufferSync|bufmgr.c#BgBufferSync]] [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#SyncOneBuffer|bufmgr.c#SyncOneBuffer]] [[raw/postgres-12/src/backend/postmaster/checkpointer.c#AbsorbSyncRequests|checkpointer.c#AbsorbSyncRequests]].

**Commit and WAL.** A transaction that did not acquire an XID and has no WAL-logged work can skip a WAL commit record [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]]. For WAL-logged committing work, `RecordTransactionCommit` inserts a commit record and, unless asynchronous commit rules apply, flushes WAL with `XLogFlush` [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]]. WAL insertion builds records in WAL buffers with `XLogInsert`, while `XLogFlush` / `XLogWrite` write WAL pages and fsync according to the configured WAL sync method [[raw/postgres-12/src/backend/access/transam/xloginsert.c#XLogInsert|xloginsert.c#XLogInsert]] [[raw/postgres-12/src/backend/access/transam/xlog.c#XLogFlush|xlog.c#XLogFlush]] [[raw/postgres-12/src/backend/access/transam/xlog.c#XLogWrite|xlog.c#XLogWrite]] [[raw/postgres-12/src/backend/access/transam/xlog.c#issue_xlog_fsync|xlog.c#issue_xlog_fsync]].

**DDL and maintenance.** Relation creation opens storage, creates relation files, and emits storage-create WAL [[raw/postgres-12/src/backend/catalog/storage.c#RelationCreateStorage|storage.c#RelationCreateStorage]] [[raw/postgres-12/src/backend/catalog/storage.c#log_smgrcreate|storage.c#log_smgrcreate]] [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrcreate|smgr.c#smgrcreate]]. Relation unlink/truncate paths drop buffers and unlink or truncate physical files [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrdounlink|smgr.c#smgrdounlink]] [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrtruncate|smgr.c#smgrtruncate]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdunlinkfork|md.c#mdunlinkfork]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdtruncate|md.c#mdtruncate]]. `TRUNCATE` emits an `XLOG_HEAP_TRUNCATE` record [[raw/postgres-12/src/backend/commands/tablecmds.c#ExecuteTruncate|tablecmds.c#ExecuteTruncate]]. `ANALYZE` samples relation data and updates `pg_statistic` rows, but ordinary first execution of a query does not update `pg_statistic`; stats rows are inserted or updated by `update_attstats` during analyze [[raw/postgres-12/src/backend/commands/analyze.c#update_attstats|analyze.c#update_attstats]]. `COPY FROM` is not a bypass of heap/index storage: it routes accepted rows through `table_tuple_insert` and then index insertion where needed [[raw/postgres-12/src/backend/commands/copy.c#CopyFrom|copy.c#CopyFrom]].

### Azure Slow-I/O Interpretation

If the hot heap and index pages for a statement are already in `shared_buffers`, those particular page accesses do not wait for Azure data-file reads; `ReadBuffer_common` returns on the shared-buffer hit before `smgrread` [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#ReadBuffer_common|bufmgr.c#ReadBuffer_common]]. Storage can still appear in latency through:

- cold relcache/catcache lookups that scan system catalogs [[raw/postgres-12/src/backend/utils/cache/catcache.c#SearchCatCacheMiss|catcache.c#SearchCatCacheMiss]];
- planner relation-size and segment operations [[raw/postgres-12/src/backend/storage/smgr/md.c#mdnblocks|md.c#mdnblocks]];
- heap/index buffer misses [[raw/postgres-12/src/backend/storage/smgr/md.c#mdread|md.c#mdread]];
- temp spills from sort or hash join [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileDumpBuffer|buffile.c#BufFileDumpBuffer]];
- dirty victim writes, bgwriter/checkpointer writes, and fsync processing [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#BufferAlloc|bufmgr.c#BufferAlloc]] [[raw/postgres-12/src/backend/postmaster/checkpointer.c#AbsorbSyncRequests|checkpointer.c#AbsorbSyncRequests]];
- WAL write/fsync at commit for synchronous WAL-logged work [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]] [[raw/postgres-12/src/backend/access/transam/xlog.c#XLogFlush|xlog.c#XLogFlush]].

### `track_io_timing` Measurement Boundary

The quoted claim is directionally right, but it needs sharper edges for PostgreSQL 12.

`track_io_timing` is a superuser-settable (`PGC_SUSET`) boolean, defaulting off. A superuser can enable it for a session without restart; changing the configured default follows the normal reload path for a non-postmaster GUC default [[raw/postgres-12/src/backend/utils/misc/guc.c#track_io_timing|guc.c#track_io_timing]].

For relation-buffer reads, the timer starts only after PostgreSQL has failed to satisfy the request from shared or local buffers. In `ReadBuffer_common`, an already-found buffer returns through the hit path before any `smgrread` call or I/O timer; the timed block starts immediately before `smgrread`, stops immediately after it, then adds the elapsed time to `pgStatBlockReadTime` and `pgBufferUsage.blk_read_time` [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#ReadBuffer_common|bufmgr.c#ReadBuffer_common]] [[raw/postgres-12/src/include/pgstat.h#pgstat_count_buffer_read_time|pgstat.h#pgstat_count_buffer_read_time]]. Therefore a shared-buffer hit increments hit/read counters but contributes no `blk_read_time`; an operating-system cache hit can still show up as a timed read because PostgreSQL has already missed `shared_buffers` and entered `smgrread`.

For relation-buffer writes, the timed path is narrower than "all writes by the statement." `FlushBuffer` first performs WAL-before-data safety work with `XLogFlush` for permanent buffers, then starts the `track_io_timing` clock just before `smgrwrite`, stops it just after `smgrwrite`, and adds the elapsed time to `pgStatBlockWriteTime` and `pgBufferUsage.blk_write_time` [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#FlushBuffer|bufmgr.c#FlushBuffer]] [[raw/postgres-12/src/include/pgstat.h#pgstat_count_buffer_write_time|pgstat.h#pgstat_count_buffer_write_time]]. That means the page-write timing excludes the preceding WAL flush in `FlushBuffer` itself. It also excludes relation-extension writes in the `P_NEW` path, where `ReadBuffer_common` calls `smgrextend` without surrounding it with the `track_io_timing` clock; `mdextend` still performs a `FileWrite` with a `DataFileExtend` wait event, but it is not added to `blk_write_time` by this code path [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#ReadBuffer_common|bufmgr.c#ReadBuffer_common]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdextend|md.c#mdextend]].

The metadata-open/lseek part is the most nuanced. Metadata work that happens outside the timed `smgrread` / `smgrwrite` windows is not measured as block I/O time. Planner relation-size probes go through `smgrnblocks`, `mdnblocks`, `_mdnblocks`, and `FileSize`, and `FileSize` uses `lseek(..., SEEK_END)`; those relation-size probes are not wrapped by the buffer-manager `track_io_timing` blocks [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrnblocks|smgr.c#smgrnblocks]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdnblocks|md.c#mdnblocks]] [[raw/postgres-12/src/backend/storage/smgr/md.c#_mdnblocks|md.c#_mdnblocks]] [[raw/postgres-12/src/backend/storage/file/fd.c#FileSize|fd.c#FileSize]]. However, if segment lookup, VFD reopening, or file access happens inside the timed `smgrread` / `smgrwrite` call, that elapsed wall-clock time is inside the measured span because the timer wraps the storage-manager call, not only the final `pg_pread` or `pg_pwrite` syscall [[raw/postgres-12/src/backend/storage/smgr/md.c#mdread|md.c#mdread]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdwrite|md.c#mdwrite]] [[raw/postgres-12/src/backend/storage/smgr/md.c#_mdfd_getseg|md.c#_mdfd_getseg]] [[raw/postgres-12/src/backend/storage/file/fd.c#PathNameOpenFile|fd.c#PathNameOpenFile]].

Temporary file I/O is counted but not timed by `track_io_timing` in PostgreSQL 12. `BufFileLoadBuffer` and `BufFileDumpBuffer` call `FileRead` / `FileWrite` and increment `pgBufferUsage.temp_blks_read` / `temp_blks_written`, but they do not use `track_io_timing` and do not add to `blk_read_time` or `blk_write_time` [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileLoadBuffer|buffile.c#BufFileLoadBuffer]] [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileDumpBuffer|buffile.c#BufFileDumpBuffer]]. This matters for slow storage: a spilling sort or hash join can show temp blocks and high latency while `I/O Timings` stays low or absent.

WAL I/O is separate from buffer I/O timing. Commit can call `XLogFlush`; `XLogWrite` writes WAL with `pg_pwrite`, and `issue_xlog_fsync` performs the configured WAL sync method, but these paths are not the `ReadBuffer_common` / `FlushBuffer` `track_io_timing` counters [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]] [[raw/postgres-12/src/backend/access/transam/xlog.c#XLogFlush|xlog.c#XLogFlush]] [[raw/postgres-12/src/backend/access/transam/xlog.c#XLogWrite|xlog.c#XLogWrite]] [[raw/postgres-12/src/backend/access/transam/xlog.c#issue_xlog_fsync|xlog.c#issue_xlog_fsync]]. They can expose wait events such as WAL write or WAL sync while the wait is happening, but they do not become foreground plan `blk_*_time` [[raw/postgres-12/src/include/pgstat.h#pgstat_report_wait_start|pgstat.h#pgstat_report_wait_start]].

Foreground query instrumentation is also process-scoped. `EXPLAIN` prints `I/O Timings` from the plan's `BufferUsage` when `track_io_timing` is enabled, and `pg_stat_statements` stores buffer-usage deltas from executor and utility hooks into its per-statement counters [[raw/postgres-12/src/backend/commands/explain.c#show_buffer_usage|explain.c#show_buffer_usage]] [[raw/postgres-12/contrib/pg_stat_statements/pg_stat_statements.c#pgss_ExecutorEnd|pg_stat_statements.c#pgss_ExecutorEnd]] [[raw/postgres-12/contrib/pg_stat_statements/pg_stat_statements.c#pgss_ProcessUtility|pg_stat_statements.c#pgss_ProcessUtility]] [[raw/postgres-12/contrib/pg_stat_statements/pg_stat_statements.c#pgss_store|pg_stat_statements.c#pgss_store]]. Parallel query workers have their buffer usage accumulated back into the leader's instrumentation, but bgwriter and checkpointer dirty-page writes are separate background processes rather than plan-node work for the foreground query [[raw/postgres-12/src/backend/executor/instrument.c#InstrAccumParallelQuery|instrument.c#InstrAccumParallelQuery]] [[raw/postgres-12/src/backend/storage/buffer/bufmgr.c#BgBufferSync|bufmgr.c#BgBufferSync]] [[raw/postgres-12/src/backend/postmaster/checkpointer.c#AbsorbSyncRequests|checkpointer.c#AbsorbSyncRequests]]. `pg_stat_database` exposes accumulated `blk_read_time` and `blk_write_time` from stats messages for a database, but those columns are still populated from the same `pgStatBlockReadTime` / `pgStatBlockWriteTime` counters, not from every possible filesystem or WAL wait [[raw/postgres-12/src/backend/catalog/system_views.sql#pg_stat_database|system_views.sql#pg_stat_database]] [[raw/postgres-12/src/backend/postmaster/pgstat.c#pgstat_send_tabstat|pgstat.c#pgstat_send_tabstat]] [[raw/postgres-12/src/backend/postmaster/pgstat.c#pgstat_recv_tabstat|pgstat.c#pgstat_recv_tabstat]] [[raw/postgres-12/src/backend/utils/adt/pgstatfuncs.c#pg_stat_get_db_blk_read_time|pgstatfuncs.c#pg_stat_get_db_blk_read_time]].

### Statement Type Summary

**`SELECT`.** Reads can happen during analysis/planning through relation and catalog lookup, during planning through relation-size probes, and during execution through heap/index/visibility-map/TOAST buffer access [[raw/postgres-12/src/backend/parser/parse_relation.c#parserOpenTable|parse_relation.c#parserOpenTable]] [[raw/postgres-12/src/backend/optimizer/util/plancat.c#estimate_rel_size|plancat.c#estimate_rel_size]] [[raw/postgres-12/src/backend/access/heap/heapam.c#heapgetpage|heapam.c#heapgetpage]] [[raw/postgres-12/src/backend/access/index/indexam.c#index_getnext_slot|indexam.c#index_getnext_slot]]. Writes are not guaranteed, but visibility checks can dirty pages through hint bits, and sort/hash work can spill to temporary files [[raw/postgres-12/src/backend/access/heap/heapam_visibility.c#SetHintBits|heapam_visibility.c#SetHintBits]] [[raw/postgres-12/src/backend/storage/file/buffile.c#BufFileDumpBuffer|buffile.c#BufFileDumpBuffer]].

**`INSERT` and `COPY FROM`.** Accepted rows go through normal table insertion; heap, index, TOAST, constraint, and sequence-related reads can occur as needed, and heap/index pages are dirtied by the insert path [[raw/postgres-12/src/backend/commands/copy.c#CopyFrom|copy.c#CopyFrom]] [[raw/postgres-12/src/backend/access/heap/heapam.c#heap_insert|heapam.c#heap_insert]] [[raw/postgres-12/src/backend/executor/execIndexing.c#ExecInsertIndexTuples|execIndexing.c#ExecInsertIndexTuples]]. Logged work emits WAL and may need commit-time WAL flush depending on synchronous commit rules [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]].

**`UPDATE`.** The executor reads the old tuple and whatever indexes or TOAST data the plan and update need; `heap_update` can dirty old/new heap pages, write TOAST data, and insert new index entries unless HOT rules avoid index changes [[raw/postgres-12/src/backend/access/heap/heapam.c#heap_update|heapam.c#heap_update]] [[raw/postgres-12/src/backend/access/heap/heapam.c#log_heap_update|heapam.c#log_heap_update]]. WAL and commit flush behavior follow the same transaction rules as other WAL-logged DML [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]].

**`DELETE`.** The delete path reads the heap tuple and relevant indexes used to find it, modifies the heap tuple state, may delete TOAST data, and leaves index tuple removal to later vacuum processing [[raw/postgres-12/src/backend/executor/nodeModifyTable.c#ExecDelete|nodeModifyTable.c#ExecDelete]] [[raw/postgres-12/src/backend/access/heap/heapam.c#heap_delete|heapam.c#heap_delete]]. Heap-delete WAL and commit flushing apply when the transaction writes WAL and synchronous commit rules require it [[raw/postgres-12/src/backend/access/transam/xact.c#RecordTransactionCommit|xact.c#RecordTransactionCommit]].

**DDL and `TRUNCATE`.** These statements can touch catalogs, dependencies, relcache state, storage metadata, and relation files. Relation create/unlink/truncate paths create, remove, or truncate physical storage and emit storage or heap-truncate WAL as applicable [[raw/postgres-12/src/backend/catalog/storage.c#RelationCreateStorage|storage.c#RelationCreateStorage]] [[raw/postgres-12/src/backend/catalog/storage.c#log_smgrcreate|storage.c#log_smgrcreate]] [[raw/postgres-12/src/backend/storage/smgr/smgr.c#smgrtruncate|smgr.c#smgrtruncate]] [[raw/postgres-12/src/backend/storage/smgr/md.c#mdtruncate|md.c#mdtruncate]] [[raw/postgres-12/src/backend/commands/tablecmds.c#ExecuteTruncate|tablecmds.c#ExecuteTruncate]].

**`VACUUM` and `ANALYZE`.** VACUUM scans heap pages and can modify heap, index, visibility-map, and free-space-map state; ANALYZE samples table data and updates `pg_statistic` rows [[raw/postgres-12/src/backend/access/heap/vacuumlazy.c#lazy_scan_heap|vacuumlazy.c#lazy_scan_heap]] [[raw/postgres-12/src/backend/access/heap/vacuumlazy.c#lazy_vacuum_heap|vacuumlazy.c#lazy_vacuum_heap]] [[raw/postgres-12/src/backend/access/heap/vacuumlazy.c#lazy_vacuum_index|vacuumlazy.c#lazy_vacuum_index]] [[raw/postgres-12/src/backend/access/heap/visibilitymap.c#visibilitymap_set|visibilitymap.c#visibilitymap_set]] [[raw/postgres-12/src/backend/storage/freespace/freespace.c#RecordPageWithFreeSpace|freespace.c#RecordPageWithFreeSpace]] [[raw/postgres-12/src/backend/commands/analyze.c#update_attstats|analyze.c#update_attstats]].

**`COPY TO` and `COPY (SELECT)`.** The query side follows the normal plan read path, while server-file output is separate from heap/index relation-buffer reads and is not made durable through the heap/index WAL path [[raw/postgres-12/src/backend/commands/copy.c#DoCopy|copy.c#DoCopy]].

## Open Questions

- Azure storage latency, host filesystem cache behavior, and kernel writeback policy are outside the PostgreSQL source tree and were not used as evidence here. The source supports where PostgreSQL can issue reads, writes, opens, truncates, and fsyncs; it does not determine a specific Azure SKU's latency.
- The exact I/O count for a real statement depends on schema, plan shape, relation and catalog cache warmth, shared-buffer contents, operating-system cache state, dirty-buffer pressure, temp spills, checksums, `wal_log_hints`, `synchronous_commit`, triggers, foreign keys, extensions, and FDWs. Those are runtime inputs, not fixed properties of the statement keyword.
