# Plan cache mode production impact (unverified)

## Question

In PostgreSQL 12, how do the different `plan_cache_mode` values affect a production environment? What is the best mode for different scenarios, with pros and cons? Include the impact of slow random I/O storage.

## Answer

Assumption: "PostgreSQL 12" means the local source checkout pinned to `45b88269a353ad93744772791feb6d01bc7e1e42`.

The production default should usually remain `plan_cache_mode = auto`. In PG 12 this GUC exists to override the prepared-statement choice between generic plans and custom plans, not to change planning for every ordinary one-shot statement. The source defines three values: `auto`, `force_generic_plan`, and `force_custom_plan` [[raw/postgres-12/src/include/utils/plancache.h#PlanCacheMode|plancache.h#PlanCacheMode]], and the GUC is `PGC_USERSET` with default `PLAN_CACHE_MODE_AUTO` [[raw/postgres-12/src/backend/utils/misc/guc.c#plan_cache_mode|guc.c#plan_cache_mode]]. `PGC_USERSET` means use session or transaction scope for production experiments; no restart or reload is required for `SET` / `SET LOCAL`. Changing a config-file or role/database default changes future defaults, but the plan-cache decision itself is made when a cached plan is executed, not when it is prepared [[raw/postgres-12/doc/src/sgml/config.sgml#guc-plan-cache_mode|config.sgml#guc-plan-cache_mode]].

`auto` is the balanced mode because PG 12 first uses custom plans for a parameterized prepared statement, records their estimated costs, then compares a generic plan against the average custom-plan cost. The code forces custom plans until `num_custom_plans < 5`, then chooses generic when `generic_cost < avg_custom_cost`; custom-plan cost includes an added planning-cost estimate [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#cached_plan_cost|plancache.c#cached_plan_cost]]. The PREPARE documentation describes the same five-execution heuristic and explains the tradeoff: generic plans save planning time, while custom plans can use supplied parameter values [[raw/postgres-12/doc/src/sgml/ref/prepare.sgml#sql-prepare-notes|prepare.sgml#sql-prepare-notes]].

## Recommendation Matrix

| Mode | Best production scenarios | Pros | Cons |
|---|---|---|---|
| `auto` | Default for mixed workloads, most application-driver prepared statements, most PL/pgSQL SQL statements, and cases where parameter skew is unknown. | Lets PG 12 learn from the first five custom executions, then reuses a generic plan only when its estimated cost beats the average custom-plan cost including planning work [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]]. It also avoids manual global choices based on a few examples. | The heuristic compares planner estimates, not observed runtime. If estimates are wrong, the generic plan can be worse in practice. The regression test intentionally shows `auto` reaching a "really bad plan" after five executions with a skewed table: the rare value gets a generic seq scan with `$1` instead of a custom index-only scan with `a = 2` [[raw/postgres-12/src/test/regress/sql/plancache.sql#plan_cache_mode|plancache.sql#plan_cache_mode]], [[raw/postgres-12/src/test/regress/expected/plancache.out#plan_cache_mode|plancache.out#plan_cache_mode]]. |
| `force_custom_plan` | Parameter-sensitive prepared statements: highly skewed predicates, tenant/status/date filters where selectivity varies widely, queries where constants unlock better index or partition choices, and slow-random-I/O systems where one bad random-heavy plan is very expensive. | Forces `choose_custom_plan()` to build a plan with the actual parameter values for each execution [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]]. PG 12 marks protocol, SQL PREPARE, SPI, and PL/pgSQL parameters as `PARAM_FLAG_CONST`, allowing the planner to substitute parameter values into a custom plan [[raw/postgres-12/src/include/nodes/params.h#PARAM_FLAG_CONST|params.h#PARAM_FLAG_CONST]], [[raw/postgres-12/src/backend/optimizer/util/clauses.c#eval_const_expressions_mutator|clauses.c#eval_const_expressions_mutator]]. | Replans every execution, so planning CPU, catalog access, planner memory churn, and planning latency rise. `GetCachedPlan()` builds a custom plan with `boundParams` whenever the policy says custom, and accumulates custom-plan cost per execution [[raw/postgres-12/src/backend/utils/cache/plancache.c#GetCachedPlan|plancache.c#GetCachedPlan]]. |
| `force_generic_plan` | Very high call-count statements where parameter values do not change the best plan, uniform lookup/update patterns, simple OLTP statements where planning CPU dominates, or temporary triage when a bad custom-plan estimate is causing unstable choices. | Skips the five-custom-plan warmup for parameterized cached plans because the force-generic check happens before the `num_custom_plans < 5` rule [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]]. A valid generic plan is stored on the `CachedPlanSource` and reused after `CheckCachedPlan()` validates it and acquires execution locks [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#GetCachedPlan|plancache.c#GetCachedPlan]]. | Hides parameter values from planning, so it can choose a plan that is only average for the workload. If the ideal plan changes by parameter value, this is the riskiest mode. The PG 12 test suite demonstrates a rare-value query where forced generic uses a seq scan with `Filter: (a = $1)` while forced custom uses an index-only scan with `Index Cond: (a = 2)` [[raw/postgres-12/src/test/regress/expected/plancache.out#plan_cache_mode|plancache.out#plan_cache_mode]]. |

Use targeted overrides rather than a blanket cluster-wide default. Because `plan_cache_mode` is `PGC_USERSET`, a production-safe pattern is to set it for one session, one transaction, one role, one database default, or one application connection pool class after measuring [[raw/postgres-12/src/backend/utils/misc/guc.c#plan_cache_mode|guc.c#plan_cache_mode]]. For PL/pgSQL specifically, PG 12 prepares a statement on first use with `SPI_prepare_params()`, keeps the plan when requested, and executes it through SPI, so function bodies can be affected even when the application did not issue SQL `PREPARE` directly [[raw/postgres-12/src/pl/plpgsql/src/pl_exec.c#exec_prepare_plan|pl_exec.c#exec_prepare_plan]], [[raw/postgres-12/src/pl/plpgsql/src/pl_exec.c#exec_stmt_execsql|pl_exec.c#exec_stmt_execsql]].

## `CheckCachedPlan()` Deep Dive

`CheckCachedPlan()` is the fast-path gate for reusing an already-built generic plan. It does not decide whether generic planning is desirable; `choose_custom_plan()` makes that policy decision first. The call sequence in `GetCachedPlan()` is: revalidate the analyzed/rewrite tree with `RevalidateCachedQuery()`, choose custom versus generic with `choose_custom_plan()`, and, only on the generic path, call `CheckCachedPlan()` to see whether `plansource->gplan` can be reused [[raw/postgres-12/src/backend/utils/cache/plancache.c#GetCachedPlan|plancache.c#GetCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]].

The function's input is a `CachedPlanSource`; the generic plan itself is the `CachedPlanSource.gplan` reference-counted link. The fields that matter to this gate are `CachedPlanSource.is_valid`, `CachedPlanSource.gplan`, and the `CachedPlan` fields `stmt_list`, `is_valid`, `planRoleId`, `dependsOnRole`, `saved_xmin`, and `refcount` [[raw/postgres-12/src/include/utils/plancache.h#CachedPlanSource|plancache.h#CachedPlanSource]], [[raw/postgres-12/src/include/utils/plancache.h#CachedPlan|plancache.h#CachedPlan]].

Step by step:

1. `CheckCachedPlan()` asserts that `plansource->is_valid` is already true. That assertion is the boundary between query-tree revalidation and plan reuse: `RevalidateCachedQuery()` already handled search-path mismatch, RLS rewrite-environment changes, parse-analysis/rewrite rebuilding, and planner-lock acquisition before `CheckCachedPlan()` is allowed to run [[raw/postgres-12/src/backend/utils/cache/plancache.c#RevalidateCachedQuery|plancache.c#RevalidateCachedQuery]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]].
2. If `plansource->gplan` is `NULL`, it returns `false`; `GetCachedPlan()` then builds a new generic plan with `BuildCachedPlan(..., boundParams = NULL, ...)`, links it into `plansource->gplan`, increments its refcount for the parent link, records `generic_cost`, and may still reject it in favor of a custom plan if the newly known generic cost loses to the custom-plan heuristic [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#GetCachedPlan|plancache.c#GetCachedPlan]].
3. If a generic plan exists, `CheckCachedPlan()` checks role validity. `BuildCachedPlan()` stores `planRoleId = GetUserId()` and sets `dependsOnRole` from RLS rewrite dependency or from any `PlannedStmt.dependsOnRole`; foreign-scan planning can set that flag when a pushdown decision assumes the current user [[raw/postgres-12/src/backend/utils/cache/plancache.c#BuildCachedPlan|plancache.c#BuildCachedPlan]], [[raw/postgres-12/src/backend/optimizer/plan/createplan.c#create_foreignscan_plan|createplan.c#create_foreignscan_plan]], [[raw/postgres-12/src/backend/optimizer/plan/planner.c#standard_planner|planner.c#standard_planner]]. If the cached plan depends on role and the current user differs from `planRoleId`, `CheckCachedPlan()` marks the plan invalid instead of reusing it [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]].
4. If the plan still appears valid, `CheckCachedPlan()` acquires executor locks by walking each `PlannedStmt` and locking relation RTEs with their recorded `rellockmode`. It also handles utility statements that contain a parsed query. This is the race-closing step: the function acquires the locks needed to run the cached plan, then rechecks whether invalidation callbacks marked the plan invalid while locks were being acquired [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#AcquireExecutorLocks|plancache.c#AcquireExecutorLocks]].
5. After locking, transient plans get one more freshness check. Planner setup initializes `PlannerGlobal.transientPlan = false`; index metadata can mark a plan transient when an index is valid but not yet safely usable because of `indcheckxmin`; `standard_planner()` copies the global flag into `PlannedStmt.transientPlan`; and `BuildCachedPlan()` saves `TransactionXmin` into `CachedPlan.saved_xmin` when any planned statement is transient [[raw/postgres-12/src/backend/optimizer/plan/planner.c#standard_planner|planner.c#standard_planner]], [[raw/postgres-12/src/backend/optimizer/util/plancat.c#get_relation_info|plancat.c#get_relation_info]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#BuildCachedPlan|plancache.c#BuildCachedPlan]]. If `TransactionXmin` has advanced since `saved_xmin`, `CheckCachedPlan()` marks the cached generic plan invalid [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]].
6. If the plan remains valid after role, lock/race, and transient-xmin checks, `CheckCachedPlan()` returns `true` with executor locks held. `GetCachedPlan()` then reuses `plansource->gplan`, increments the active reference count, and returns it to the caller [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#GetCachedPlan|plancache.c#GetCachedPlan]].
7. If the plan becomes invalid after executor locks were acquired, `CheckCachedPlan()` releases those locks, unlinks `plansource->gplan` through `ReleaseGenericPlan()`, and returns `false`. `ReleaseGenericPlan()` sets `plansource->gplan = NULL` before calling `ReleaseCachedPlan()`, so the next generic path must build a fresh generic plan [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#ReleaseGenericPlan|plancache.c#ReleaseGenericPlan]].

The invalidation callbacks explain why `CheckCachedPlan()` can fail even after the query tree was valid. Relcache invalidation can mark the query tree and generic plan invalid when the rewritten query depends on a relation, and it can also mark only the generic plan invalid when the generic plan has extra relation dependencies beyond the query tree [[raw/postgres-12/src/backend/utils/cache/plancache.c#PlanCacheRelCallback|plancache.c#PlanCacheRelCallback]]. The object-cache callback does the same for function/type invalidation items, and broader namespace/operator/access-method/FDW/syscache invalidations call `ResetPlanCache()` through `PlanCacheSysCallback()` [[raw/postgres-12/src/backend/utils/cache/plancache.c#PlanCacheObjectCallback|plancache.c#PlanCacheObjectCallback]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#PlanCacheSysCallback|plancache.c#PlanCacheSysCallback]].

Production consequences:

- A valid generic plan is cheap but not free: every reuse still passes through `CheckCachedPlan()` and relation lock acquisition for the plan's relation RTEs. For a simple OLTP statement this is usually much cheaper than replanning; for a generic plan with many referenced relations or partitions, the lock walk is part of the per-execution cost of choosing `force_generic_plan` or of `auto` settling on generic reuse [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#AcquireExecutorLocks|plancache.c#AcquireExecutorLocks]].
- Generic-plan invalidation creates latency spikes on the next execution. `CheckCachedPlan()` itself only returns `false`; `GetCachedPlan()` then builds a replacement generic plan, records its cost, and may build a custom plan instead if the generic is now estimated to lose [[raw/postgres-12/src/backend/utils/cache/plancache.c#GetCachedPlan|plancache.c#GetCachedPlan]].
- Changing `plan_cache_mode` does not by itself flush old generic plans; the GUC entry has no assign hook, and the function is consulted at execution time [[raw/postgres-12/src/backend/utils/misc/guc.c#plan_cache_mode|guc.c#plan_cache_mode]], [[raw/postgres-12/doc/src/sgml/config.sgml#guc-plan-cache_mode|config.sgml#guc-plan-cache_mode]]. Setting `force_custom_plan` bypasses `CheckCachedPlan()` because `choose_custom_plan()` returns custom, but setting `force_generic_plan` can reuse an already-valid `plansource->gplan` if one exists [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]].
- Planner cost GUC changes are similar: `seq_page_cost` and `random_page_cost` are `PGC_USERSET` planner-cost GUCs with no assign hook, so changing them in a session does not by itself make `CheckCachedPlan()` reject an already-valid generic plan [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesReal|guc.c#planner-cost-gucs]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#CheckCachedPlan|plancache.c#CheckCachedPlan]]. When testing slow-random-I/O cost settings against generic plans, deallocate/reprepare the statement or use `DISCARD PLANS`; `DEALLOCATE` drops the `CachedPlanSource`, and `DISCARD PLANS` calls `ResetPlanCache()` [[raw/postgres-12/src/backend/commands/prepare.c#DropPreparedStatement|prepare.c#DropPreparedStatement]], [[raw/postgres-12/src/backend/commands/discard.c#DiscardCommand|discard.c#DiscardCommand]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#ResetPlanCache|plancache.c#ResetPlanCache]].

## Boundaries And Edge Cases

- No parameters: PG 12 never needs a custom plan when `boundParams == NULL`, so a parameterless prepared statement uses a generic plan regardless of the mode [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]].
- One-shot cached plans: `choose_custom_plan()` treats one-shot plans as custom before it looks at `plan_cache_mode`, so the GUC cannot make one-shot planning reusable [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]].
- Named vs. unnamed extended-query protocol: named statements are stored through `StorePreparedStatement()`, while an unnamed statement is dropped when the next unnamed parse arrives. A client that parses/binds/executes unnamed statements once per query gives the generic plan little opportunity to amortize planning work [[raw/postgres-12/src/backend/tcop/postgres.c#exec_parse_message|postgres.c#exec_parse_message]], [[raw/postgres-12/src/backend/tcop/postgres.c#drop_unnamed_stmt|postgres.c#drop_unnamed_stmt]].
- DDL, search path, RLS, role-dependent plans, relation invalidations, and some catalog invalidations can force revalidation or invalidate a generic plan. `plancache.c` tracks saved plan sources, reacquires locks, and invalidates query trees or generic plans on relevant relcache/syscache events [[raw/postgres-12/src/backend/utils/cache/plancache.c#RevalidateCachedQuery|plancache.c#RevalidateCachedQuery]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#PlanCacheRelCallback|plancache.c#PlanCacheRelCallback]].
- PG 12's `pg_prepared_statements` view exposes name, statement, prepare time, parameter types, and `from_sql`; it does not expose generic/custom execution counters in this checkout [[raw/postgres-12/src/backend/commands/prepare.c#pg_prepared_statement|prepare.c#pg_prepared_statement]], [[raw/postgres-12/src/backend/catalog/system_views.sql#pg_prepared_statements|system_views.sql#pg_prepared_statements]]. Use `EXPLAIN EXECUTE` to distinguish plans: generic output keeps `$n` parameters, while custom output shows substituted parameter values [[raw/postgres-12/doc/src/sgml/ref/prepare.sgml#sql-prepare-notes|prepare.sgml#sql-prepare-notes]].

## Slow Random I/O Impact

Slow random I/O does not change the `plan_cache_mode` algorithm directly; it magnifies the cost of a wrong plan choice. PG 12's planner cost model has explicit `seq_page_cost` and `random_page_cost` variables, expects sequential page cost to be lower than random page cost, and lets these costs be overridden per tablespace when different data lives on different storage [[raw/postgres-12/src/backend/optimizer/path/costsize.c#costsize|costsize.c#costsize]]. A seq scan charges sequential page cost for table pages [[raw/postgres-12/src/backend/optimizer/path/costsize.c#cost_seqscan|costsize.c#cost_seqscan]], while index-scan heap access charges random page cost for uncorrelated page fetches and blends toward sequential cost only when correlation is high [[raw/postgres-12/src/backend/optimizer/path/costsize.c#cost_index|costsize.c#cost_index]].

For mechanical disks or cloud block devices with poor random latency, first make the cost model honest. PG 12 documents `random_page_cost` default `4.0`, explains that this assumes many random reads are cached, says raising `random_page_cost` makes index scans look more expensive, and recommends increasing it when the assumed cache rate is too optimistic [[raw/postgres-12/doc/src/sgml/config.sgml#guc-random-page-cost|config.sgml#guc-random-page-cost]]. `seq_page_cost`, `random_page_cost`, and `effective_cache_size` are `PGC_USERSET`, so session/transaction tests need no restart or reload; config/default changes affect later sessions/defaults [[raw/postgres-12/src/backend/utils/misc/guc.c#ConfigureNamesReal|guc.c#planner-cost-gucs]], [[raw/postgres-12/src/backend/utils/misc/guc.c#effective_cache_size|guc.c#effective_cache_size]].

Mode guidance under slow random I/O:

| Scenario on slow random I/O | Preferred mode | Reason |
|---|---|---|
| Uniform parameters and one stable access path | `auto`, or targeted `force_generic_plan` after proving the generic plan matches the custom plan shape. | Generic reuse saves planning CPU. Slow disks do not matter much if the same plan is correct for every parameter. |
| Skewed predicates where rare and common values need different access paths | `auto` first, then targeted `force_custom_plan` for the bad statement if `auto` settles on the wrong generic plan. | The wrong plan is more expensive when random page misses are slow. The PG 12 regression test shows exactly the class of failure: custom planning uses a rare-value index-only scan, while generic planning uses a broader seq scan with `$1` [[raw/postgres-12/src/test/regress/expected/plancache.out#plan_cache_mode|plancache.out#plan_cache_mode]]. |
| Broad parameters can produce many random heap fetches through an index or nested loop | Prefer `force_custom_plan` only when per-value planning avoids that random-I/O-heavy path; otherwise fix cost constants/statistics first. | This is an inference from the cost model: uncorrelated index heap fetches are charged with `random_page_cost`, and `choose_custom_plan()` compares estimated costs rather than measured latency [[raw/postgres-12/src/backend/optimizer/path/costsize.c#cost_index|costsize.c#cost_index]], [[raw/postgres-12/src/backend/utils/cache/plancache.c#choose_custom_plan|plancache.c#choose_custom_plan]]. |
| Bitmap heap scans are chosen and the storage can handle concurrent I/O | Tune `effective_io_concurrency`; do not use `plan_cache_mode` as the fix. | PG 12 documents `effective_io_concurrency` as the number of concurrent I/O operations a session may initiate, currently affecting bitmap heap scans only; higher values can help SSDs or disk arrays but can add CPU overhead if higher than needed [[raw/postgres-12/doc/src/sgml/config.sgml#guc-effective-io-concurrency|config.sgml#guc-effective-io-concurrency]], [[raw/postgres-12/src/backend/executor/nodeBitmapHeapscan.c#ExecInitBitmapHeapScan|nodeBitmapHeapscan.c#ExecInitBitmapHeapScan]]. |

The slow-disk rule of thumb is: do not use `force_generic_plan` as a blanket latency fix unless the generic and custom plan shapes are proven equivalent across representative parameters. Slow random I/O makes parameter-sensitive mistakes more visible, so it often increases the value of targeted `force_custom_plan` for the few statements where parameter values really decide whether random I/O is acceptable.

## Production Probe Pattern

For a SELECT-only prepared statement, compare modes in one transaction. Use session-scoped timeouts before the probe; `statement_timeout`, `lock_timeout`, and `plan_cache_mode` are all `PGC_USERSET`, so these `SET LOCAL` changes need no restart or reload [[raw/postgres-12/src/backend/utils/misc/guc.c#statement_timeout|guc.c#statement_timeout]], [[raw/postgres-12/src/backend/utils/misc/guc.c#lock_timeout|guc.c#lock_timeout]], [[raw/postgres-12/src/backend/utils/misc/guc.c#plan_cache_mode|guc.c#plan_cache_mode]]. The 30s statement timeout and 1s lock timeout are conservative starting points for a read-only plan probe; shorten them for latency-critical systems.

```sql
BEGIN /* wiki_plan_cache_mode_probe */;
SET /* wiki_plan_cache_mode_probe */ LOCAL statement_timeout = '30s';
SET /* wiki_plan_cache_mode_probe */ LOCAL lock_timeout = '1s';

SET /* wiki_plan_cache_mode_probe */ LOCAL plan_cache_mode = force_generic_plan;
EXPLAIN /* wiki_plan_cache_mode_probe */ (ANALYZE, BUFFERS, TIMING OFF)
EXECUTE prepared_statement_name(...);

SET /* wiki_plan_cache_mode_probe */ LOCAL plan_cache_mode = force_custom_plan;
EXPLAIN /* wiki_plan_cache_mode_probe */ (ANALYZE, BUFFERS, TIMING OFF)
EXECUTE prepared_statement_name(...);

ROLLBACK /* wiki_plan_cache_mode_probe */;
```

For DML prepared statements, avoid `EXPLAIN (ANALYZE)` in production unless side effects are acceptable inside a transaction and rollback is enough for the specific statement. Use plain `EXPLAIN EXECUTE` in production and run `EXPLAIN (ANALYZE, BUFFERS)` on staging with production-like data.

## Open Questions

- The PG 12 source does not encode a universal "best" `plan_cache_mode`; the recommendation matrix is an operational synthesis from source-visible behavior, cost model inputs, and regression tests. Production choice still needs measurement on representative parameters and storage.
- The pinned PG 12 tree does not expose per-statement generic/custom execution counters in `pg_prepared_statements`; persistent monitoring of plan choice requires plan sampling, application-side experiment tracking, or external instrumentation.
